Prolog in and JSON out


The Prolog programming language plays no less than four different roles in SWAPP: as a query language, as a rule language, as a server-side web application programming language, and as the language in which the platform as a whole is implemented. But when SWAPP talks back to clients, only JavaScript Object Notation (JSON) is used.

Prolog has a number of traits that distinguishes it from most other programming languages:

  • Declarativeness, inherited from its roots in logic
  • Inference capabilities, from its roots in theorem proving
  • A-tuple-at-a-time generation of solutions to queries
  • No separation between data and programs (reflectivity)
  • Only one kind of simple datastructure – terms

Its declarative nature and its inference capabilities places Prolog closer to the Semantic Web than most other general programming languages. RDF graphs may be queried through Prolog queries formally equivalent to SPARQL. Rules increase the expressiveness and serves as a bridge to future Semantic Web technologies. But Prolog is also a general programming language, which implies that the interaction between queries and application code becomes seamless, something which tends to be import­ant in practice.

The a-tuple-at-a-time generation of solutions comes in handy when we prefer to generate the solutions one-by-one, perhaps for reasons of memory require­ments (on server and/or client), or perhaps just because we want to be able to decide, after having seen the first couple of solutions, whether we want to see more.

Reflectivity is the key to safety. Providing access to a general programming language over the web could be dangerous. Reflectivity means that we can easily inspect programs and queries before running them, denying attempts to execute code that might jeopardize the integrity of the server. Since terms – the only kind of datastructures available in Prolog – are so simple, this is done with less effort than in most other languages.

Our approach to remote procedure call (RPC) also employs the reflective capabilities of the Prolog language to invoke procedures remotely. This simplifies the programming model since a procedure does not have to be declared in any way to receive remote calls. As long as calls are safe and clients are authorized, remote procedure calls will execute just as easily as queries, and there is no essential difference whatsoever.

The results of queries and events coming from the systems, while terms on the server-side, are converted into JSON before being returned to the client.
